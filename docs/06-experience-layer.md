# experience/ 使用指南

## 两级学习

Harness 的经验积累分两级：

| 级别 | 触发方式 | 内容 | 维护者 |
|------|----------|------|--------|
| **低级：自动** | Agent 每次执行后写入 | progress.tsv、report 中的 FAIL 记录 | Agent 自动 |
| **高级：人工策展** | 用户定期 review 后提炼 | patterns.md、failures.md | 人工 |

低级学习是原材料，高级学习是提炼后的知识。
**不要跳过人工策展** —— agent 自动记录的信息量大但噪声也大。

## patterns.md 格式

```markdown
# 成功模式

## P-001: API 错误处理统一模式
- **发现日期**: 2025-03-15
- **场景**: 所有 HTTP 客户端的错误处理
- **模式**: 使用 Result<T, ApiError> 包装，不用 try-catch
- **收益**: QA 通过率从 60% 提升到 90%
- **适用范围**: 所有涉及外部 API 调用的模块

## P-002: 组件测试优先于页面测试
- **发现日期**: 2025-04-02
- **场景**: 前端功能验证
- **模式**: 先测单个组件的 props 和事件，再测页面集成
- **收益**: Bug 定位时间从 20min 降到 5min
- **适用范围**: 所有前端组件开发
```

## failures.md 格式

```markdown
# 失败记录

## F-001: Mock 与真实 API 返回格式不一致
- **发现日期**: 2025-03-20
- **影响**: 472 个测试全通过但生产环境崩溃
- **根因**: Mock 使用 { data: ... }，真实 API 返回 { result: ... }
- **修复**: 添加 API Schema 验证到 CI
- **预防措施**: QA 维度中添加"mock 一致性检查"
- **复发次数**: 0（已预防）

## F-002: 全局搜索遗漏导致 Whack-a-Mole
- **发现日期**: 2025-04-10
- **影响**: 修 1 个文件后发现 3 个同类 bug
- **根因**: Builder 未执行全局审计
- **修复**: Builder prompt 添加审计指令
- **预防措施**: progress.tsv 中必须有"审计范围"列
- **复发次数**: 1（需强化）
```

## 反馈到 Prompt 的流程

```
1. Agent 执行产出 report（自动）
         │
         ▼
2. 人工 review report，发现重复模式（人工）
         │
         ▼
3. 提炼到 patterns.md 或 failures.md（人工）
         │
         ▼
4. 将关键模式写入 Agent prompt（人工）
   例：Builder prompt 添加 "修复 bug 后必须全局搜索同类问题"
         │
         ▼
5. 下次执行时 Agent 自动遵循（自动）
```

**关键洞见**：经验不是自动变成 prompt 的。
中间必须有人工策展步骤来过滤噪声、提炼原则。

## 前沿信号监控

定期检查这两个指标：

### 指标 1：接纳率

```
接纳率 = QA 首次通过的任务数 / 总任务数
```

- > 80%：Harness 运转良好
- 60%-80%：Builder prompt 需要优化，检查 failures.md 中的高频问题
- < 60%：可能是 spec 质量问题，检查 Coordinator 的意图翻译

### 指标 2：首轮成功率

```
首轮成功率 = 不需要返工的任务数 / 总任务数
```

- > 70%：优秀
- 50%-70%：正常（复杂任务往往需要 1-2 轮迭代）
- < 50%：检查 spec 的颗粒度是否太粗

## 人工策展的重要性

Agent 自动记录的 progress.tsv 和 report 文件会快速增长。
如果不定期策展：

- patterns.md 为空，agent 不断重复已知的好实践
- failures.md 为空，agent 不断踩已知的坑
- prompt 越来越长（塞了太多未提炼的原始记录）

**建议节奏**：每 10 个任务做一次策展。
读最近的 report 文件，提炼 1-2 条到 patterns.md 或 failures.md。
同时清理过时的记录。

**记住：experience/ 的价值不在于记录多少，在于提炼了多少。**
