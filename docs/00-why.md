# 为什么需要 Harness Infra

## 问题：单 Agent 的三个结构性瓶颈

用 Claude Code 完成一个 80 行的函数重构，体验很好。
但让它接手一个"重构认证模块"的任务，你大概率会撞进这三堵墙：

### 瓶颈 1：你在做反馈环

Agent 写完代码后说"已完成"。你打开浏览器手动检查，发现少了 edge case，
回来告诉它，它修好了又引入新问题。**你变成了反馈循环的人力齿轮。**

问题不在于 agent 能力不够，而在于没有一个独立角色专门负责验收。
你承担了 Evaluator 角色却没有系统化的检查清单。

### 瓶颈 2：没有跨任务学习

上周修了一个 Date format 的 bug，这周同样的 bug 出现在另一个模块。
Agent 的 context 只在单次会话内存活，**没有持久化的经验积累。**

### 瓶颈 3：意图和执行混在一起

你说"优化首页加载速度"，agent 立刻开始改代码。
但"优化"可以是懒加载、可以是 CDN、可以是图片压缩 ——
**没有 spec 锚点，改到一半才发现方向不对。**

## 解决方案：三角色分离 + 自动化反馈环

```
意图层   [Planner/Coordinator]  →  spec.md（锚点）
执行层   [Generator/Builder]    →  代码 / 文档（产物）
验收层   [Evaluator/QA]         →  report（反馈）
         ↑_________反馈环_________↓
```

三件事同时解决：

| 瓶颈 | 对应机制 |
|------|----------|
| 你在做反馈环 | Evaluator agent 自动验收，人只在 escalation 时介入 |
| 没有跨任务学习 | `.harness/experience/` 持久化模式和失败记录 |
| 意图执行混合 | Planner 先产出 spec，Generator 只能按 spec 执行 |

## 不解决什么

诚实声明 —— Harness 不能帮你：

- **做架构决策**：它是流程基础设施，不替你想清楚该怎么设计
- **替代领域知识**：如果 CLAUDE.md 里没写清楚项目约束，Harness 也救不了
- **保证零 bug**：Evaluator 的覆盖范围取决于你写的 QA 维度
- **管理团队协作**：这是单人 + AI 的工作流，不是团队 Scrum 工具

## 为什么是"基础设施"不是"框架"

框架有运行时、有依赖、有 API。Harness 没有。

它的全部实体是：
- 几个 `.md` 文件（agent 定义）
- 一个 `settings.json`（路由配置）
- 一个 `.harness/` 文件夹（中间产物）

删掉 `.harness/` 和 `.claude/agents/`，你的项目零影响。
这才是"基础设施"——**可以随时铺设、随时拆除，不和你的代码耦合。**

## 谁该用

- 已经在用 Claude Code，想让复杂任务更可靠
- 任务涉及多文件改动、需要验收环节
- 希望 AI 辅助流程可审计、可回溯
- 团队多人共用 Claude Code 需要统一规范

## 谁不该用

- 只做单次问答（Claude Code 直接用就好）
- 寻找 LangChain / AutoGen 替代品（这不是代码库）
- 任务本身 10 分钟能做完（Harness 的 overhead 不划算）
