# Harness 随模型升级精简

## 核心洞见

> **Harness 中的每个组件都在编码"模型做不到"的假设。**
> 当模型能力提升，对应的组件应该被精简或移除。

Harness 不是越复杂越好。它的理想状态是随模型进步越来越薄，
最终只剩下少数"不随模型升级而过时"的核心机制。

## 历史对照

### 虚构示例：Sprint 分解的兴衰

**2024 年初（200k context）**

模型 context window 只有 200k。一个涉及 15 个文件的重构任务，
spec 必须拆成 3 个 sprint，每个 sprint 只改 5 个文件。
Coordinator 要管理 sprint 之间的依赖和交接。

Harness 配置包含：
- `sprint-plan.md`：Sprint 分解模板
- `handoff.md`：Sprint 间交接清单
- Coordinator prompt 中 40 行的 sprint 管理指令

**2025 年（1M context）**

模型 context window 扩展到 1M。同样的 15 文件重构，
一次会话就能看全所有文件。Sprint 分解变成了纯 overhead。

精简操作：
- 删除 `sprint-plan.md` 模板
- 删除 `handoff.md` 模板
- Coordinator prompt 中 sprint 管理指令从 40 行减到 0
- 改为单次 spec，不再分 sprint

**节省**：每次任务减少约 15 分钟的 sprint 管理开销。

### 更多可能精简的组件

| 组件 | 编码的假设 | 何时精简 |
|------|-----------|----------|
| 详细的 progress.tsv | 模型长会话会忘记做了什么 | 当模型可靠维持全会话记忆 |
| spec 中的文件级分解 | 模型无法自行判断影响范围 | 当模型自主全局审计足够可靠 |
| QA 的逐条 checklist | 模型验收不够系统化 | 当模型自主验收达到人类水平 |
| 多轮循环上限 | 模型可能陷入无效循环 | 当模型自主识别收敛/发散 |

## 精简检查清单

每次模型大版本升级后，逐条检查：

- [ ] **Sprint 分解**：新 context 长度是否让 sprint 分解成为 overhead？
- [ ] **Progress 追踪**：模型是否能可靠回忆本次会话的所有操作？
- [ ] **Spec 粒度**：模型是否能从高层 spec 自行分解到文件级？
- [ ] **QA 细度**：模型自主验收的 PASS/FAIL 准确率是否足够？
- [ ] **循环控制**：模型是否能自主判断"继续修"还是"escalate"？
- [ ] **Handoff 模板**：跨会话恢复是否还需要显式交接文件？

**每个 YES 意味着对应的 Harness 组件可以精简或移除。**

## 精简的方法

不是直接删除，而是渐进退化：

```
阶段 1：保留组件，但在 prompt 中标记为可选
        "如果你能完整记住本次所有操作，可以跳过 progress.tsv 更新"

阶段 2：观察 3-5 个任务，确认跳过后没有质量下降

阶段 3：移除组件，更新模板
```

## 什么永远不该精简

无论模型能力如何提升，以下机制不应移除：

### 1. 三角色分离

Planner ≠ Generator ≠ Evaluator 不是因为模型"做不到"，
而是因为**同一个实体不应该自评自己的产出**。
这是系统设计原则，不是模型能力的补丁。

即使模型达到 AGI 水平，自评偏差仍然存在。
人类专家也需要独立审计，AI 也一样。

### 2. 文件审计层

`.harness/` 中的 spec、progress、report 文件不只是给 agent 看的，
也是给人看的。可审计性和可回溯性是流程价值，不是模型补丁。

### 3. 经验持久化

跨会话的学习（patterns.md、failures.md）解决的是
"会话间知识丢失"问题。即使模型有完美记忆，
新会话仍然从零开始。经验文件是跨会话的桥梁。

## 思维模型

```
Harness 组件 = 核心机制 + 模型补丁

核心机制：角色分离 + 审计层 + 经验持久化
         └─ 永远保留

模型补丁：Sprint 分解 + 详细 progress + 循环上限 + ...
         └─ 随模型升级逐步精简
```

**目标：让 Harness 越来越薄，直到只剩下核心机制。**
